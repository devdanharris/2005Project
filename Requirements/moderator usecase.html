Title: Accepting Users to Private Channels
Primary Actor: Moderator
Stakeholders and Interests:

Moderator - wants to filter users who want to gain access into the collaborative tool.
User - trying to gain access into the collaborative tool.

Precondition: User with Moderator priviledges logged in already, and has access to MUN's registration database.
Trigger: Moderator notified of new user trying to gain access into SuD.
Main success scenario:

  1. Moderator selects view user requests action.
  2. SuD lists unreviewed user requests.
  3. Moderator selects specific user to review from list.
  4. SuD lists user's Credentials as given.
  5. Moderator checks to make sure user is formally registered as a student at MUN or if the user is a special guest.
  6. Moderator changes users request to accepted.
  7. SuD records user accepted, reviewer, time and date.
  8. SuD notifies user of change in status
  9. Moderator repeats 3-9 as desired.

Extensions:

  1. 4a. User put in credentials in the wrong format:
    1. 4a1. SuD notifies user of wrong credentials.
    2. 4a2. User re-enters credentials correctly
    3. 4a3. User initiates Accepting Users use case
  2. 5a. User is not a formally registered student or a special guest:
    1. 5a1. Moderator checks user's credentials.
    2. 5a2. Moderator rejects students user's to join the collaborative tool
    3. 5a3. SuD notifies student of rejection
  3. 5b. Database mderator uses to check student's credentials is down:
    1. 5b1. Moderator tries to verify student's enrollment.
    2. 5b2. Moderator waits till server is back on
    3. 5b3. SuD notifies student of extra wait time in gaining access.
